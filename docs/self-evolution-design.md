# 自进化模式设计（2026-06-10 落地）

## 三层架构

```
┌─ 监督层（agentic，LLM=Claude /loop）────────────────────┐
│  审查扫描结果 / 对比前向 vs 回测 / 提出新策略家族假设 /  │
│  发现系统性问题（如卖出信号负期望）→ 改引擎             │
├─ 调度层（确定性，server 内置）──────────────────────────┤
│  evolve_sweep：全宇宙(56标的)顺序进化，自适应窗口        │
│  QHH_AUTOSWEEP_HOURS=24 每日自动扫描（0=关）             │
│  POST /api/evolve_sweep 手动触发 / GET .../status        │
├─ 引擎层（确定性，无LLM）────────────────────────────────┤
│  (μ+λ) GA，5策略家族（含RuleVote技术规则投票）           │
│  4折CPCV-lite适应度 + 留出闸门（margin 0.1 + floor 0)    │
│  晋升 → 冠军注册表 → 模拟盘自动重置开新会话              │
└──────────────────────────────────────────────────────────┘
```

## Agent 要不要参与？——分层回答

- **引擎/调度层不需要 LLM**：进化、闸门、晋升、模拟盘全是确定性代码，
  LLM 进来只会引入不可复现性。
- **监督层需要**：哪些假设值得做成新家族（如 RuleVote 的诞生）、负结果归因
  （如仓位调制 A/B 失败）、前向与回测漂移的诊断——这些是 LLM 的活，
  通过 /loop 周期唤醒执行，产出写 LOOP_STATE.md 留痕。

## 正反馈定义（验收）

1. ✅ 回测闸门通过率：首轮扫描 56 标的 → 41 晋升（73%），
   拒绝了全部留出≤0的标的（BTC/ETH/CEG/HOOD/QCOM/VST 等）——闸门在工作
2. ✅ 跨家族竞争有效：RuleVote 3 个纯冠军（AMD holdout 2.41 全场最高）+
   大量 ensemble 内含 rule_vote 成员
3. ⏳ 前向验证（最终裁判）：44 个模拟盘会话从 2026-06-10 起算，
   监督层每轮唤醒对比组合净值 vs 回测预期，4 周后出首份前向报告
4. ⏳ 自动扫描稳定性：每日 auto-sweep 不应频繁换冠军
   （margin=0.1 抑制噪音晋升；若 churn 过高 → 升 margin）

## 风险与对策

- **44会话×回测乐观偏差**：留出 Sharpe 是单次评估，跨标的同期相关
  （科技股一起跌时 44 个会话一起回撤）→ 组合层 15% 熔断已有；
  监督层盯组合均值净值
- **每日扫描数据窥探**：同一留出窗反复被挑战者看到 → 后续做滚动留出
  （窗口随日期前移，自然防止重复用同一段）
- **短历史标的**（ARM/CRWV/GEV/SNDK <1100根bar）：跳过，攒够再入池
